# 多智能体结构化辩论协议

## 目的

把外部文章/知识研究从“单模型理解笔记”升级为“多视角辩论后的验证性理解”。

适用场景：
- 高影响文章，可能影响本仓库 Pattern / Protocol 的默认规则
- 单模型理解后仍存在明显不确定性、矛盾或高风险主张
- 用户明确要求深度验证而不是直接吸收

## 在工作流中的位置

```text
文章入库 → 单模型理解 → 多智能体辩论 → Pattern 候选 → 验证 → 固化
```

本协议位于：
- 理解笔记之后
- Pattern 固化之前

## 最低配置

### 三个基础角色

| 角色 | 核心问题 | 输出重点 |
|---|---|---|
| 实践工程师 | 这在我们的环境里能用吗？ | 可用性、成本、维护 |
| 怀疑论者 | 这有证据吗？哪里逻辑跳跃？ | 证据、利益冲突、反例 |
| 学术研究者 | 这有什么理论基础？ | 文献先例、验证状态、rigor |

### 执行原则

1. 三个角色必须**隔离上下文**独立分析
2. 不允许互看对方结论后再写
3. 必须先有一版单模型理解笔记，作为对照基线
4. 综合时必须显式列出：
   - 矛盾点
   - 共识点
   - 单模型遗漏洞察
   - 同源共同盲区

## 输出结构

```markdown
## 矛盾点
## 共识点
## 单模型遗漏洞察
## 同源共同盲区
## 是否值得回写本仓库
```

## 成功判据

当以下至少两项成立时，说明辩论优于单模型：

- 发现单模型完全遗漏的洞察
- 识别出原文或单模型中的逻辑矛盾
- 提供更严格的适用性判断
- 给出证据等级区分（作者声称 / 文献支持 / 我推断）

## 局限

- 同源模型的多子代理仍然共享底层偏见
- 因此本协议建议在高价值研究中加入“异质模型补充轮”
- 异质模型补充适合在 Obsidian 真源中记录，但不应直接升级为默认规则，除非再次验证

## 与本仓库的关系

本协议是一个**候选研究协议**，不是强制默认。只有经过多次文章实验和至少一次真实项目验证后，才考虑提升为通用 Pattern 或强制流程。
